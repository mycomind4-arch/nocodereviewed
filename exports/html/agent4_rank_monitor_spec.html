Agent 4

Rank Monitor

Daily position tracking across 10 sites and all tracked keywords via the Google Search Console API. Instant Slack/email alerts on drops >5 positions. Full history in Supabase. Weekly mover digest every Monday.

Daily trigger — 06:00 UTC GSC Search Analytics API Supabase rank_history Slack + Email alerts Weekly digest — Monday 07:00 UTC
📐 What this agent does
  1. Triggers daily at 06:00 UTC (after GSC data refreshes)
  2. Loops over all 10 sites × all tracked keywords
  3. Fetches yesterday's average position from GSC Search Analytics API
  4. Fetches the position from 2 days ago out of Supabase for comparison
  5. If drop > 5 positions → fires Slack alert + email immediately
  6. Writes today's position record to rank_history table
  7. On Mondays at 07:00 UTC → runs weekly digest of biggest movers
🗂 Prerequisites
  • All 10 sites verified in Google Search Console under a single Google account (or a service account with access to each property)
  • GSC OAuth 2.0 credentials created in Google Cloud Console (project: nocode-empire)
  • Supabase project running with rank_history table (see Schema section)
  • A tracked_keywords table already populated (covered in schema section)
  • Slack workspace with a #seo-alerts channel and incoming webhook configured
  • SMTP or Resend credentials for email fallback
  • n8n instance (self-hosted or n8n.cloud) with credentials store access
GSC data latency: GSC Search Analytics data has a 2–3 day lag. Scheduling the workflow at 06:00 UTC and querying for date = today - 2 days (not yesterday) gives you the freshest complete dataset. Querying for yesterday will return incomplete or zero rows for most sites.
Setup

Authentication Setup

Agent 4 needs two auth paths: a service account for unattended GSC access, and a Supabase service key for DB writes.

Google Cloud — Service Account

A service account is preferable to OAuth refresh tokens for an automated agent because it never expires and requires no manual re-authentication.

Step 1 — Create the project & enable APIs

Google Cloud Console
# 1. Create or select a project
gcloud projects create nocode-empire-seo --name="No-Code Empire SEO"
gcloud config set project nocode-empire-seo

# 2. Enable the Search Console API
gcloud services enable searchconsole.googleapis.com

# 3. Create the service account
gcloud iam service-accounts create rank-monitor \
  --display-name="Rank Monitor Agent"

# 4. Create & download the JSON key
gcloud iam service-accounts keys create rank-monitor-key.json \
  --iam-account=rank-monitor@nocode-empire-seo.iam.gserviceaccount.com

Step 2 — Grant access to each GSC property

In Google Search Console UI: for each of the 10 sites, go to Settings → Users and permissions → Add user. Enter the service account email and set permission to Full.

Service account email format
rank-monitor@nocode-empire-seo.iam.gserviceaccount.com
Domain properties vs URL-prefix properties: If your sites are set up as Domain properties in GSC (e.g. sc-domain:example.com), the service account must be added as a user to each domain property individually — domain properties don't inherit access from the Google account owner.

Step 3 — Add credentials to n8n

In n8n, go to Credentials → New → Google Service Account. Paste the entire contents of rank-monitor-key.json into the Service Account Key field. Name it GSC Rank Monitor. This credential is referenced by the HTTP Request nodes in the daily workflow.

Data layer

Supabase Schema

Two tables: tracked_keywords (the source list) and rank_history (the append-only log). A view wires them together for the weekly digest query.

rank_history — Primary log table

SQL — run in Supabase SQL Editor
CREATE TABLE rank_history (
  id          BIGSERIAL     PRIMARY KEY,
  keyword     TEXT          NOT NULL,
  site        TEXT          NOT NULL,   -- GSC siteUrl, e.g. 'sc-domain:example.com'
  position    NUMERIC(6,2)  NOT NULL,   -- GSC average position (can be fractional)
  date        DATE          NOT NULL,   -- The data date (not insert date)
  clicks      INTEGER       DEFAULT 0,
  impressions INTEGER       DEFAULT 0,
  ctr         NUMERIC(5,4)  DEFAULT 0,  -- e.g. 0.0312 = 3.12%
  inserted_at TIMESTAMPTZ   DEFAULT NOW(),

  CONSTRAINT rank_history_unique
    UNIQUE (keyword, site, date)       -- idempotent upserts safe
);

-- Indexes for the queries we run most
CREATE INDEX idx_rh_site_date
  ON rank_history (site, date DESC);

CREATE INDEX idx_rh_keyword_site
  ON rank_history (keyword, site);

CREATE INDEX idx_rh_date
  ON rank_history (date DESC);

tracked_keywords — Source of truth

CREATE TABLE tracked_keywords (
  id          BIGSERIAL    PRIMARY KEY,
  keyword     TEXT         NOT NULL,
  site        TEXT         NOT NULL,   -- matches rank_history.site
  priority    TEXT         DEFAULT 'normal', -- 'high' | 'normal' | 'low'
  target_url  TEXT,         -- the page you want ranking for this keyword
  active      BOOLEAN      DEFAULT TRUE,
  created_at  TIMESTAMPTZ  DEFAULT NOW(),

  CONSTRAINT tk_unique UNIQUE (keyword, site)
);
Populating tracked_keywords: Insert one row per (keyword, site) pair. Start with your 20–30 highest-priority keywords per site. The daily workflow reads this table at runtime — adding or removing rows immediately affects what Agent 4 monitors, no workflow edits needed.

weekly_movers view

This view powers the Monday digest by self-joining rank_history to compare last 7 days vs the prior 7 days.

CREATE VIEW weekly_movers AS
WITH
  this_week AS (
    SELECT keyword, site,
           AVG(position) AS avg_pos_this
    FROM   rank_history
    WHERE  date >= CURRENT_DATE - 7
      AND  date <  CURRENT_DATE
    GROUP BY keyword, site
  ),
  prior_week AS (
    SELECT keyword, site,
           AVG(position) AS avg_pos_prior
    FROM   rank_history
    WHERE  date >= CURRENT_DATE - 14
      AND  date <  CURRENT_DATE - 7
    GROUP BY keyword, site
  )
SELECT
  t.keyword,
  t.site,
  ROUND(t.avg_pos_this::numeric, 1)   AS pos_this_week,
  ROUND(p.avg_pos_prior::numeric, 1)  AS pos_prior_week,
  ROUND((p.avg_pos_prior - t.avg_pos_this)::numeric, 1) AS delta  -- positive = improved
FROM  this_week t
JOIN  prior_week p USING (keyword, site)
ORDER BY ABS(delta) DESC;

RLS policy

-- Enable RLS on both tables
ALTER TABLE rank_history       ENABLE ROW LEVEL SECURITY;
ALTER TABLE tracked_keywords   ENABLE ROW LEVEL SECURITY;

-- Service role bypasses RLS automatically (no policy needed for it)
-- Block anon reads from rank_history (optional but recommended)
CREATE POLICY "No anon access"
  ON rank_history
  FOR ALL
  TO anon
  USING (false);

GSC API Calls

Exact request structure for the Search Analytics: query endpoint. One call per site per run — the keyword filter brings back all tracked keywords in a single request.

Endpoint

FieldValue
MethodPOST
URLhttps://searchconsole.googleapis.com/webmasters/v3/sites/{siteUrl}/searchAnalytics/query

Request body

This single request fetches all tracked keywords for one site in one call, returning per-query rows. The dimensionFilterGroups filters to only the keywords in your tracked list — no need to loop per keyword.

JSON body — constructed dynamically in n8n Code node
{
  "startDate":  "2025-05-20",   // TODAY - 4 days (safest for complete data)
  "endDate":    "2025-05-20",   // same as startDate — single day snapshot
  "dimensions": ["query"],
  "rowLimit":   25000,
  "dimensionFilterGroups": [{
    "filters": [{
      "dimension":  "query",
      "operator":   "includingRegex",
      "expression": "best zapier alternatives|no code automation tools|..."
      // pipe-separated regex of all tracked keywords for this site
      // built dynamically from tracked_keywords table
    }]
  }]
}
Why includingRegex instead of per-keyword calls? The GSC API allows max 200 API calls per 100 seconds per user. With 10 sites × 50 keywords = 500 calls/day if you loop per keyword. Using includingRegex collapses this to 10 calls total — one per site. The regex pipe limit is well above 50 keywords per site.

Response structure

{
  "rows": [
    {
      "keys":        ["best zapier alternatives"],
      "clicks":      42,
      "impressions": 1840,
      "ctr":          0.0228,
      "position":    4.3     // average position for that day
    },
    // ... one row per keyword that had impressions that day
  ]
}
Zero-impression keywords: If a keyword had no impressions on the target date, GSC omits it from the response entirely. The n8n Code node must handle missing keywords by setting position = null (not 0) in the Supabase insert. A null position means "not seen," not "position 0."

Code Node (JavaScript)
Build per-site GSC request objects
Groups keywords by site. For each site, builds the regex string and request body. Outputs one item per site. 
const rows = $input.all();
const targetDate = new Date();
targetDate.setDate(targetDate.getDate() - 4);  // D-4 for complete data
const dateStr = targetDate.toISOString().slice(0,10);

// Group by site
const bySite = {};
for (const row of rows) {
  const s = row.json.site;
  if (!bySite[s]) bySite[s] = [];
  bySite[s].push(row.json.keyword);
}

return Object.entries(bySite).map(([site, keywords]) => ({
  json: {
    siteUrl: site,
    keywords,
    targetDate: dateStr,
    requestBody: {
      startDate: dateStr,
      endDate: dateStr,
      dimensions: ["query"],
      rowLimit: 25000,
      dimensionFilterGroups: [{
        filters: [{
          dimension: "query",
          operator: "includingRegex",
          expression: keywords.map(k => k.replace(/[.*+?^${}()|[\]\\]/g, '\\$&')).join('|')
        }]
      }]
    }
  }
}));

Code Node (JavaScript)
Diff positions, flag drops >5
Merges GSC response with prior-day lookup. Computes delta. Flags any keyword where newPos - prevPos > 5 (higher number = worse rank). 
const gscRows  = $node["GSC Query"].json.rows || [];
const prevData  = $node["Fetch Prev Positions"].json;
const prevMap   = Object.fromEntries(prevData.map(r => [r.keyword, r.position]));
const siteUrl   = $node["Build Requests"].json.siteUrl;
const today     = $node["Build Requests"].json.targetDate;

const results = [];
for (const row of gscRows) {
  const kw      = row.keys[0];
  const newPos  = row.position;
  const prevPos = prevMap[kw] ?? null;
  const delta   = prevPos !== null ? newPos - prevPos : null;
  results.push({
    json: {
      keyword: kw, site: siteUrl,
      position: newPos, prevPosition: prevPos,
      delta, date: today,
      clicks: row.clicks, impressions: row.impressions,
      ctr: row.ctr,
      alert: delta !== null && delta > 5
    }
  });
}
return results;

Weekly Digest Workflow

Runs every Monday at 07:00 UTC. Queries the weekly_movers view and sends a ranked digest of top gainers and losers across all 10 sites.

Workflow name: Agent4_RankMonitor_WeeklyDigest — Schedule trigger: 07:00 UTC every Monday
①
Schedule Trigger
Weekly trigger — Monday 07:00 UTC
Cron: 0 7 * * 1
②
HTTP Request → Supabase RPC
Query weekly_movers view
GET {{SUPABASE_URL}}/rest/v1/weekly_movers?select=*&order=delta.asc&limit=50
This returns all movers sorted by delta. The next node splits into gainers (delta > 0) and losers (delta < 0).
③
Code Node
Build digest payload
Splits rows into gainers (delta < 0, improved) and losers (delta > 0, worsened). Takes top 10 of each. Formats Slack blocks. 
const rows = $input.all().map(r => r.json);
const gainers = rows.filter(r => r.delta < 0)
  .sort((a,b) => a.delta - b.delta)
  .slice(0,10);          // most negative delta = biggest gain
const losers  = rows.filter(r => r.delta > 0)
  .sort((a,b) => b.delta - a.delta)
  .slice(0,10);          // most positive delta = biggest loss

return [{ json: { gainers, losers } }];

First run expectation: On day 1, there are no "previous" positions in Supabase. The diff node will produce prevPosition = null and alert = false for all keywords — no false alerts will fire. Day 2 onwards, comparisons work normally. After 7 days, the weekly digest view has enough data to produce meaningful movers.
GSC data gap days: If a site has zero GSC impressions on the target date (new site, or a holiday weekend anomaly), the API returns an empty rows array. The Code node guards for this with || []. Nothing gets written to Supabase for that date — intentional. The weekly digest uses AVG so a missing day doesn't skew the numbers.
