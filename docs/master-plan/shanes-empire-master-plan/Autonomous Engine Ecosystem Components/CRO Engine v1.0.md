## SECTION 01 — What This Is

The Monetization Engine scores every page's revenue efficiency and surfaces gaps. The Audience Engine explains which reader cohorts are engaging with which content. What neither engine does is systematically change anything on the page and measure whether the change improved conversion.

Without testing, the stack can identify that a page is underperforming but cannot learn what would make it perform better. Every optimisation decision — where to place a CTA, how to write a verdict, whether a pricing table improves conversion — is made by assumption. Some assumptions are right. Many are wrong. None are ever confirmed.

This engine closes that loop. It selects pages worth testing, designs variants based on what the existing data suggests might be wrong, runs controlled tests with proper statistical handling, measures outcomes against the Audience Engine's established baselines, and applies winning variants automatically. It then feeds what it learned back into the Content Engine so future articles are generated with proven patterns rather than assumptions.

### What the other specs do not cover

- Systematically varying page elements and measuring the effect on affiliate conversion rate
- Distinguishing between a page that converts poorly because of the offer and one that converts poorly because of the content structure
- Learning which CTA positions, article structures, and verdict formats drive the highest conversion rates for each content type
- Building a library of proven patterns that the Content Engine applies to new article generation
- Providing statistical confidence that an observed performance difference is real rather than noise

> **CRITICAL:** This engine requires the Audience Engine to have been running for at least 2 weeks before any test can be run. The Audience Engine's `audience_content_baselines` table is the control group. Without a stable baseline, there is no meaningful control to test against. Do not activate this engine immediately after installing the Audience Engine — wait for baselines to establish.

## SECTION 02 — Architecture Overview

Five layers. One test at a time per page. A growing pattern library applied to every future article.

| Layer | Name | What it does | When it runs |
|---|---|---|---|
| 1 | Select | Identifies pages that are candidates for testing based on traffic volume, RES score, and baseline stability. Prioritises tests by expected revenue impact. | Weekly, Sunday 05:30 UTC |
| 2 | Design | For each selected test, determines what element to test and generates the variant. One element per test. Variants are generated by Gemini using proven pattern templates. | Weekly, Sunday 05:45 UTC |
| 3 | Run | Serves the variant to a statistically defined split of incoming sessions. Uses server-side rendering — no client-side flicker. Logs every session to the correct test group. | Continuous during test period |
| 4 | Measure | Checks running tests weekly for statistical significance. Stops tests that reach significance or exhaust the maximum test duration. | Weekly, Sunday 05:30 UTC (before Select) |
| 5 | Apply | Applies winning variants permanently to the page. Writes the winning pattern to the pattern library. Feeds proven patterns into the Content Engine for future generation. | On significance confirmation |

### What this engine does NOT do

- Run multiple simultaneous tests on the same page (one test per page at a time, always)
- Test structural changes that would affect Quality Gateway scores — variants are constrained to elements outside the scored content body
- Automatically discard losing variants without admin visibility — all test results are retained and visible
- Use third-party A/B testing JavaScript libraries — all test logic is server-side to avoid page speed impact

## SECTION 03 — Layer 1 — Select

Runs weekly, Sundays at 05:30 UTC — immediately after the Audience Engine's Feed layer completes at 05:00 UTC.

### Candidate qualification criteria

A page qualifies as a test candidate if it meets all four criteria:

| Criterion | Requirement | Why |
|---|---|---|
| Traffic volume | Minimum 200 sessions in the past 28 days from `audience_sessions` | Below this, tests run for months and still don't reach significance |
| Baseline stability | At least 2 weeks of data in `audience_content_baselines` with < 20% week-over-week variance in conversion rate | Unstable baselines produce false signals |
| No active test | `cro_tests.status != 'running'` for this page | One test per page at a time |
| Not recently tested | No completed test on this page in the past 60 days | Allow time for winning variants to influence organic behaviour before re-testing |

### Test prioritisation

Qualified candidates are ranked by expected revenue impact — not just by traffic volume. A page with 500 sessions and RES 15 (dead weight) is a better test candidate than a page with 500 sessions and RES 70 (already strong).

```
test_priority_score = (1 - (res_score / 100)) × session_count_28d × traffic_quality_multiplier
```

Where `traffic_quality_multiplier` is read from `audience_res_adjustments`. A high-traffic page with low RES and high-quality traffic is the highest priority test target.

The engine selects the top 3 candidates per weekly cycle and creates test records for them. If the test backlog from previous weeks already has 3+ pending tests, no new tests are created until the backlog clears.

### Test type assignment

Based on the page's RES score breakdown and Audience Engine data, the engine assigns the most likely test type:

| Condition | Test type assigned | Reasoning |
|---|---|---|
| `conversion_fit` score = 0 or low | `cta_placement` | Article isn't converting — most likely a CTA visibility or position problem |
| High engagement (scroll depth > 70%) but low conversion | `cta_copy` | Readers are consuming the content but not clicking — the CTA copy isn't compelling |
| High bounce rate from `audience_cohorts` | `headline` | Readers aren't getting past the opening — headline or intro isn't matching their expectation |
| Low scroll depth (< 40% average) | `article_structure` | Readers are leaving early — structural problem before the CTA is even seen |
| Strong content but weak offer metrics | `offer_positioning` | Content is good but the offer isn't being presented compellingly |
| pSEO page with low quality_score | `template_section` | Template element is underperforming — test a section variant |

## SECTION 04 — Layer 2 — Design

Runs weekly, Sundays at 05:45 UTC.

### Variant generation

For each newly selected test, the Design layer generates one variant using Gemini. The prompt is tightly constrained — it receives the current element, the test type, the page's audience data, and the pattern library of previously winning variants.

The Design layer uses Gemini for variant generation because the variants need to be contextually coherent with the existing article — a CTA rewrite needs to match the article's voice, a headline test needs to match the topic. Template-based variants produce generic results. Gemini-generated variants are specific to the page.

#### Variant generation prompt structure

```
You are generating a single variant for an A/B test on a content affiliate site.

PAGE CONTEXT:
- Article type: [article_type]
- Primary tool reviewed: [tool_slug]
- Current conversion rate: [X]% (site average: [Y]%)
- Primary reader cohort: [cohort_description]
- Test type: [test_type]

CURRENT ELEMENT:
[current element text/HTML]

PATTERN LIBRARY (previously winning variants for this test type):
[top 3 winning patterns from cro_pattern_library for this test_type and article_type]

CONSTRAINTS:
- Match the existing article's voice exactly
- Do not introduce any banned phrases from site_settings.banned_phrases
- Do not change the article's factual claims
- The variant must be clearly different from the control — not a minor wording tweak
- Do not add more than 50 words to the element being tested

Generate exactly one variant. Return only the variant text/HTML with no explanation.
```

One Gemini call per test design. Variants are stored in `cro_variants` with status `pending_review`.

### Admin approval gate

All variants require admin approval before the test goes live. This is not optional. The approval gate exists because:

1. Gemini occasionally produces variants that are technically different but practically meaningless
2. Some variants may conflict with sponsor agreements or brand guidelines
3. The admin should know what is being tested on the site

The TESTS tab in the admin CRO section shows all pending variants with a side-by-side comparison of the control and variant. One-click approve or reject. Rejected variants are logged with an optional rejection reason. Approved tests move to `status = running`.

> **IMPORTANT:** Tests do not start automatically. They start when the admin approves the variant. If no variants are approved in a given week, no tests run that week. The approval step takes approximately 2 minutes per test — review the control vs variant, confirm it's meaningfully different, approve. That is the only required weekly action for this engine.

## SECTION 05 — Layer 3 — Run

### Traffic splitting

Tests use a 50/50 traffic split between control and variant. Server-side rendering — the split decision is made at the edge function level before the page HTML is assembled, so there is no client-side content swap and no layout shift.

Split assignment:

```
test_group = (hash(session_token + test_id) % 100) < 50 ? 'control' : 'variant'
```

Using the session token ensures each visitor consistently sees the same version across page loads within a session. Using the test_id in the hash ensures different tests on different pages assign traffic independently.

Split assignment is logged to `cro_session_assignments` on first page load. Subsequent loads in the same session read the existing assignment.

### What gets changed server-side

The server-side rendering layer reads `cro_tests` for any running test on the requested page. If a test is running and the session is assigned to the variant group, the specific element defined in the test is swapped before the page HTML is sent to the browser. Everything else is identical.

Elements that can be swapped server-side:

| Test type | Element location | How it's swapped |
|---|---|---|
| `cta_placement` | CTA button position (in_body / see_also / end_of_article) | CSS class controls placement; class is swapped per group |
| `cta_copy` | CTA button text and surrounding sentence | Text node replacement in template |
| `headline` | H1 and meta title | String replacement in page template |
| `article_structure` | Section order (verdict first vs verdict last) | Section block reordering in template |
| `offer_positioning` | Affiliate offer presentation block | Block HTML replacement |
| `template_section` | pSEO page section (for pSEO tests only) | Section HTML replacement |

### Test duration and sample size

Minimum test duration: 14 days regardless of significance. This prevents false positives from day-of-week effects.

Maximum test duration: 56 days (8 weeks). Tests that have not reached significance after 56 days are stopped and logged as `inconclusive`.

Minimum sessions per group for significance check: 100. Tests are not evaluated for significance until both groups have at least 100 sessions.

## SECTION 06 — Layer 4 — Measure

Runs weekly, Sundays at 05:30 UTC — before Select runs.

### Primary metric

The primary metric for all tests is `affiliate_conversion_rate`: the percentage of sessions on the test page that include at least one affiliate click.

```
conversion_rate_variant = sessions_with_click (variant) / total_sessions (variant)
conversion_rate_control = sessions_with_click (control) / total_sessions (control)
relative_lift = (conversion_rate_variant - conversion_rate_control) / conversion_rate_control
```

### Statistical significance

The engine uses a two-proportion z-test for significance calculation. This runs in Supabase without a Gemini call — it is pure arithmetic on the session counts.

```
p_value = two_proportion_z_test(
  conversions_control, sessions_control,
  conversions_variant, sessions_variant
)
```

Significance threshold: p < 0.05. This means there is less than a 5% probability the observed difference is due to chance.

A test reaches a positive conclusion when:
- p < 0.05 AND
- Minimum 14 days have elapsed AND
- Both groups have at least 100 sessions AND
- Relative lift is at least +5% (filters out statistically significant but practically meaningless differences)

A test reaches a negative conclusion (variant is worse or neutral) when:
- p < 0.05 with negative lift AND minimum 14 days elapsed
- OR maximum duration (56 days) elapsed without significance

### Secondary metrics

Secondary metrics are recorded but do not determine test outcome. They provide context for interpreting results.

| Metric | Source | What it tells you |
|---|---|---|
| Average scroll depth | `audience_events` grouped by test group | Whether the variant changes how deeply readers engage |
| Average time on page | `audience_events` grouped by test group | Whether the variant speeds up or slows down the buyer journey |
| Newsletter signup rate | `audience_sessions.did_signup_newsletter` grouped by test group | Whether a CTA change affects newsletter conversion (important — a CTA that improves affiliate clicks but kills signups is a net loss) |
| Bounce rate | Single-page sessions / total sessions, grouped by test group | Whether the variant affects first impressions |

### Results stored per test

| Field | Description |
|---|---|
| sessions_control | Total sessions assigned to control group |
| sessions_variant | Total sessions assigned to variant group |
| conversions_control | Sessions with affiliate click in control group |
| conversions_variant | Sessions with affiliate click in variant group |
| conversion_rate_control | Decimal |
| conversion_rate_variant | Decimal |
| relative_lift | Decimal (positive = variant better) |
| p_value | Statistical significance |
| status | `running` / `significant_positive` / `significant_negative` / `inconclusive` / `stopped` |
| secondary_metrics | jsonb (scroll depth, time on page, signup rate, bounce rate per group) |

## SECTION 07 — Layer 5 — Apply

Runs when a test reaches significance — not on a fixed schedule.

### Applying a winning variant

When a test reaches `significant_positive` status:

1. The variant element permanently replaces the control element on the page (direct database write to `articles.body_html` or `pseo_pages.body_html` for the specific element)
2. The test is marked `status = applied`
3. The winning variant is written to `cro_pattern_library` with its test metadata
4. An alert is sent to `ALERT_EMAIL`: "[Site] CRO win: [test type] on [page title] — [relative_lift]% lift"

### Applying a losing variant

When a test reaches `significant_negative` status:

1. The control element is confirmed as the permanent state — no page change
2. The test is marked `status = applied` (control wins)
3. The losing pattern is written to `cro_pattern_library` with `outcome = loser` — these are just as valuable as winners because they prevent repeating failures
4. An alert is sent to `ALERT_EMAIL`: "[Site] CRO result: [test type] on [page title] — variant underperformed, control retained"

### The pattern library

Every test result — winner or loser — is written to `cro_pattern_library`. This table is the engine's accumulated learning. It stores:

| Field | Description |
|---|---|
| test_type | `cta_placement` / `cta_copy` / `headline` / `article_structure` / `offer_positioning` / `template_section` |
| article_type | `tool_review` / `comparison` / `tutorial` / `best_of_list` / `pseo_page` |
| control_element | The original element text/HTML |
| variant_element | The tested variant text/HTML |
| relative_lift | Observed lift (negative for losers) |
| p_value | Statistical confidence |
| sessions_in_test | Total sessions the result is based on |
| primary_traffic_source | Dominant traffic source during the test |
| outcome | `winner` / `loser` / `inconclusive` |
| site_id | |
| completed_at | |

### Feeding winners into the Content Engine

After each winning variant is applied, the Feed layer extracts the pattern and writes a structured insight to `cro_content_engine_patterns`. The Content Engine reads this table when generating new articles of the same type.

Examples of patterns fed to the Content Engine:

| Test result | Pattern written | Effect on Content Engine |
|---|---|---|
| Moving verdict to top of article improved conversion 23% | `article_structure: verdict_first = winner for tool_review` | Generation prompt for `tool_review` articles is updated to include: "Lead with verdict in first 150 words." |
| CTA copy "Try [Tool] free — no credit card" outperformed "Visit [Tool]" by 31% | `cta_copy: free_trial_emphasis = winner` | Affiliate CTA template in generation prompt updated with winning copy pattern |
| Pricing table before pros/cons improved conversion 18% for `price_checker` cohort | `article_structure: pricing_table_position = early for price_checker cohort` | Generation prompt includes pricing placement instruction for commercial-intent keywords |

Pattern application to the Content Engine is the highest-leverage output of this engine. A single CTA copy test that runs on one article produces a pattern that improves the conversion rate of every future article of that type — not just the one that was tested.

> **INFO:** Pattern application to the Content Engine prompt increments `site_settings.generation_prompt_version` automatically. The Quality Gateway's Learn layer can then correlate the prompt version change with any subsequent changes in gateway pass rates or affiliate conversion performance.

## SECTION 08 — What Can Be Tested

### Testable elements by page type

#### Content Engine articles

| Test type | What changes | What does not change |
|---|---|---|
| `cta_placement` | Position of affiliate CTA button: immediately after verdict, at end of article, in a "see also" box, or inline within the body paragraph where tool is first mentioned | Article body text, headline, meta description |
| `cta_copy` | Text of the CTA button and the surrounding sentence: "Try [Tool] free", "See [Tool] pricing", "Start with [Tool] today" | Article body, button destination URL |
| `headline` | H1 tag and meta title: compare question format ("Is [Tool] Worth It?") vs verdict format ("[Tool] Review: Best for [Use Case]") vs year format ("[Tool] Review 2026") | Article body, meta description |
| `article_structure` | Order of major sections: verdict-first vs verdict-last, pricing placement, limitations placement | Section content, headline, CTAs |
| `offer_positioning` | How the affiliate offer is framed in the article body: price-led ("From $X/month"), outcome-led ("Saves 3 hours/week"), free-trial-led ("Free plan available") | Factual accuracy of pricing claims (these still validate against tools table) |

#### pSEO pages

| Test type | What changes | What does not change |
|---|---|---|
| `template_section` | Order or emphasis of a specific template section: winner verdict position, pricing table placement, feature matrix vs pros/cons | Factual data in template slots (these come from tools table and are not changed by tests) |
| `cta_copy` | CTA button text on the primary affiliate button | Destination URL, article body |
| `headline` | H1 and meta title pattern for a page type: apply the winner to all pages of that type after test | Individual page content |

> **IMPORTANT:** pSEO page headline tests work differently from article headline tests. Because pSEO pages are template-generated, a winning headline pattern for a page type (e.g. `tool_vs_tool`) is applied to all pages of that type on the next weekly refresh, not just the tested page. This is a high-leverage test — one result improves hundreds of pages.

### What cannot be tested

- **Factual content claims** — pricing figures, feature lists, pros and cons are data from the tools table and are not subject to A/B testing
- **Quality Gateway dimensions** — tests are constrained to elements that do not change the scored content body. The gateway re-scores variant content if structural changes are involved; variants that would lower the gateway score below threshold are rejected at design time
- **Sponsored content disclosure** — legally required disclosures cannot be tested for presence/absence
- **Multiple elements simultaneously** — one element per test per page, always. Multivariate testing requires more traffic than most affiliate sites generate

## SECTION 09 — Admin Panel Additions

The Smart Admin Dashboard gains a **CRO** section in the sidebar navigation. It appears automatically when `cro_tests` table is detected. Placement: between MONETIZATION and NETWORK REPORTS.

### Four tabs in the CRO section

| Tab | What it contains |
|---|---|
| Tests | All active, pending, and recently completed tests. Each test shows: page title, test type, current sessions per group, current conversion rates, p-value, days running, status. Pending variants show control vs variant side-by-side with Approve / Reject buttons. |
| Results | All completed tests with full results: lift, significance, secondary metrics, control element, variant element. Filter by test type, article type, outcome. "Apply to Content Engine prompt" button for winners not yet applied. |
| Pattern Library | All entries in `cro_pattern_library`: winners and losers organised by test type and article type. Search by keyword. Export as JSON for backup. Shows how many articles have been generated using each winning pattern. |
| Settings | Minimum session threshold (default 200). Minimum test duration (default 14 days). Maximum test duration (default 56 days). Significance threshold (default p < 0.05). Minimum lift for positive conclusion (default 5%). Pause all tests toggle. |

## SECTION 12 — Activation Prompt

Paste this entire block into your Lovable project after the Content Engine, Monetization Engine, Audience Engine, and Smart Admin Dashboard are all installed and the Audience Engine has been running for at least 2 weeks.

> **IMPORTANT:** This prompt uses approximately 2–3 Lovable credits. The Audience Engine must have at least 2 weeks of `audience_content_baselines` data before this engine can run meaningful tests. Install this prompt after that threshold is met. It will not overwrite any existing tables, routes, or components.

---

```
Build and activate the Autonomous CRO Engine v1.0 for this Lovable project.

Before building anything, audit for: (1) audience_content_baselines table
(Audience Engine), (2) audience_sessions table, (3) monetization_res_scores table
(Monetization Engine), (4) articles table, (5) pseo_pages table,
```

## SECTION 13 — Setup — Two Questions

### Question 1 — Test aggressiveness

How many simultaneous tests do you want running across your site?

| Setting | Select | What it means |
|---|---|---|
| Conservative — 1 active test at a time | `conservative` | Only the single highest-priority candidate is tested at any time. Slower learning but cleaner data — no risk of tests interfering with each other if a user visits multiple tested pages in one session. |
| Standard — 3 active tests at a time | `standard` | Default. Top 3 candidates run simultaneously on different pages. Tests on different pages do not share sessions, so there is no cross-contamination. |
| Aggressive — 5 active tests at a time | `aggressive` | For high-traffic sites (5,000+ sessions/week). More tests running means the pattern library fills faster. Only select this if you have consistent traffic across many pages. |

### Question 2 — Pattern application scope

When a winning pattern is discovered, where should it be applied?

| Setting | Select | What it means |
|---|---|---|
| This site only | `site` | Winning patterns are applied to this site's Content Engine prompt and pSEO templates only. |
| All sites in network | `network` | Winning patterns are broadcast to all sites in `network_domains` via the `network_reports` shared table. Each site's Content Engine reads the shared pattern library and applies winners. High-value for networks — a test on Site A benefits Sites B through Z immediately. |

> **INFO:** Network pattern sharing sends the pattern instruction text only — not the specific variant HTML. Each site's Content Engine generates its own implementation of the pattern using its own voice baseline. A "verdict-first" pattern discovered on Site A instructs Site B's Gemini to lead with the verdict in its own voice, not to copy Site A's exact wording.

## SECTION 14 — Troubleshooting

> **IMPORTANT:** Before spending a Lovable credit on a fix prompt, always check `cro_test_results` and `cro_session_assignments` in Supabase directly. Most test anomalies are explained by uneven traffic split (check that session assignment hash is working correctly) or by the test starting during an unusual traffic period (holiday, viral post).

### Tests not being selected

- Check `audience_content_baselines` has at least 2 rows per page with dates more than 14 days apart — if baselines are too recent, no pages qualify
- Check `monetization_res_scores` has rows — pages without RES scores have no priority score and are not selected
- Check that no pages have `active_test_id` set to a running test — pages with active tests are excluded from selection
- If site traffic is low (< 200 sessions/page in 28 days), no pages will qualify — lower the minimum session threshold in the CRO Settings tab

### Variants not appearing (control is always served)

- Check `cro_tests.status` for the test — if it is `pending_approval`, admin has not approved it yet
- Verify the server-side rendering modification was applied to the page route — check the build report for confirmation it was added to `/blog/[slug]` and pSEO routes
- Check that the session_token is being passed correctly as a request header from the inline script
- Open Supabase and check `cro_session_assignments` — if no rows exist for recent sessions, the assignment logic is not running

### Tests running but not reaching significance after many weeks

- Check session counts per group in `cro_test_results` — if both groups have fewer than 100 sessions after 4+ weeks, the traffic volume is too low for the selected page. Stop the test and lower the minimum threshold or wait for more traffic.
- Check if the page's traffic pattern is highly variable (seasonal topic, viral spikes) — variable traffic produces unreliable significance calculations. The 14-day minimum duration helps but very spiky traffic may require 8 weeks.
- A p-value stuck above 0.3 after 500+ sessions per group means the variant is genuinely not different enough from the control to produce a detectable effect. Stop and design a more distinctive variant.

### Winning pattern not appearing in Content Engine output

- Check `cro_content_engine_patterns.active = true` for the pattern
- Verify `site_settings.generation_prompt_version` incremented when the pattern was applied — if not, the prompt update failed
- Check the master_generation_prompt in site_settings contains the pattern instruction — view it in the Content Settings tab of the admin panel
- New articles generated after the prompt version increment will use the pattern. Articles already in the queue were assigned their prompt version at queue entry time — they will not use the new pattern unless re-queued.

### pSEO pages not updating after template pattern win

- Template pattern wins are applied during the next weekly refresh run (Mondays 08:00 UTC)
- Check `cro_pseo_template_patterns` has a row for the winning pattern with the correct `pseo_page_type`
- Check the pSEO Engine refresh agent reads `cro_pseo_template_patterns` — if the Monetization Engine was installed before the CRO Engine, the refresh agent may need a targeted update: "Update pSEO weekly refresh agent to read cro_pseo_template_patterns and apply winning section HTML before regenerating pages of the matching page_type."
