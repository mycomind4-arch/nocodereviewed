# VibeCode Authority Corrections And Change Log Model

## Principle

VibeCode Authority can change verdicts as products change, but it must not hide why. Corrections, evidence refreshes, and verdict changes need a visible trail so readers can trust the current recommendation and understand what changed.

## Correction Types

| Type | Trigger | Response | Severity |
| --- | --- | --- | --- |
| Factual correction | Incorrect pricing, feature availability, limit, ownership, policy, or capability | Correct the page, attach a source snapshot, and add a dated note when the error affected a verdict | High |
| Evidence refresh | Source freshness expired, benchmark result is outdated, or official docs changed | Recheck the source, rerun affected benchmarks when needed, and update freshness state | Medium |
| Verdict change | New benchmark evidence, production blocker, pricing shift, or feature launch changes recommendation strength | Log old verdict, new verdict, evidence delta, and impacted pages before publishing | High |
| Disclosure update | Affiliate, sponsor, vendor-provided evidence, or conflict state changes | Update disclosure copy before any commercial link or sponsor placement remains live | High |
| Editorial clarification | Language is ambiguous, overbroad, unsupported, or missing use-case boundaries | Tighten claim language and record whether the change affected score, ranking, or CTA copy | Low |

## Public Policy

- Material factual errors require a dated correction note when they affected a verdict, ranking, or recommendation.
- Minor wording fixes do not require public correction unless they changed the meaning of a claim.
- Vendor requests must cite evidence; preference alone cannot change a score or verdict.
- Changed pricing, limits, exports, security posture, or policy language can invalidate affected claims immediately.
- Every verdict change must preserve the old conclusion, new conclusion, evidence delta, and changed date.

## Revision Workflow

1. Detect change through source watcher, benchmark result, QA check, vendor message, or reader report.
2. Classify correction type and severity.
3. Attach evidence: source snapshot, artifact, benchmark run, QA note, or review record.
4. Update impacted pages, tables, CTAs, reports, and social assets.
5. Publish a change note when the revision affects a verdict, ranking, recommendation, pricing claim, or disclosure.

## Change Log Fields

- Date
- Page
- Previous claim or verdict
- New claim or verdict
- Evidence delta
- Impacted rankings or CTAs
- Reviewer note
- Public correction required: yes/no

## Operating Outcome

This model turns corrections into an authority signal. The site becomes more credible when it shows how recommendations evolve as Lovable, Bolt, Replit, Cursor, Windsurf, Base44, v0, and the wider category change.
