# VibeCode Authority MVP Launch Execution Board

## Principle

The launch should move by evidence readiness, not by how finished the interface feels. The MVP can ship with provisional labels, but it cannot ship with unsupported final recommendations.

## Execution Tracks

| Track | Owner | Now | Done |
| --- | --- | --- | --- |
| Production Foundation | Engineering | Keep static app stable, define durable schema, and choose backend migration path | Public routes, admin capture, auth, database, and deployment path are working |
| Evidence Batch One | Research | Capture source snapshots and benchmark artifacts for the first seven core platforms | Every flagship review has benchmark, snapshot, artifact, review record, and QA record |
| Authority Content | Editorial | Turn completed evidence into reviews, comparisons, methodology, and first best-of pages | No page uses final winner language unless evidence parity and QA gates pass |
| Distribution and Revenue | Growth | Prepare email capture, visual assets, disclosure-safe CTAs, and launch announcements | Newsletter, affiliate disclosure, social cards, and report capture work without overclaiming |

## Launch Decision Rules

- Ship private alpha when navigation, verifier, admin capture, and core dashboards are stable.
- Ship public beta with provisional labels if the first evidence batch is incomplete but methodology is clear.
- Do not ship final rankings until each ranked page has evidence parity and QA pass.
- Do not launch affiliate CTAs until disclosure, pricing freshness, and monetization guardrails pass.
- Publish the first report only when the evidence window and exclusions are visible above the findings.

## Operating Rule

The MVP is allowed to be incomplete. It is not allowed to be vague about what is incomplete.
