# VibeCode Authority First Evidence Sprint Board

## Principle

The first launch sprint should produce proof, not drafts. A review can be beautiful later; first it needs benchmark runs, source snapshots, artifacts, review records, and QA.

## Five-Day Sprint

| Day | Focus | Output | Acceptance |
| --- | --- | --- | --- |
| Day 1 | Source snapshots | Pricing, docs, terms, export, deployment, and privacy snapshots for the first 7 tools | Every P0 tool has dated source records and blocked unknowns listed |
| Day 2 | Prompt-to-app benchmarks | Lovable, Bolt.new, Replit Agent, and Base44 run through the SaaS MVP prompt | Each run has screenshots/artifact link, deploy result, manual repair notes, and score draft |
| Day 3 | UI and IDE benchmarks | v0 landing-page task plus Cursor and Windsurf existing-codebase task | Each run records output quality, fix burden, handoff quality, and blocker notes |
| Day 4 | Review records | Seven review records with verdict state, pros, cons, production notes, and evidence gaps | No final verdict language remains where benchmark or source evidence is incomplete |
| Day 5 | QA and launch labels | Quality checks, provisional/final labels, comparison eligibility, and launch blockers | Ready pages can enter beta; blocked pages show explicit missing proof |

## Acceptance Criteria

- Every completed task links to a tool, benchmark or source type, owner, and dated record.
- Every benchmark run includes prompt, output notes, artifact, manual repair burden, and deploy/handoff state.
- Every review record declares blocked, provisional, evidence-backed, or ready verdict state.
- Every public-facing claim can map to source snapshot, benchmark run, artifact, review record, or QA check.
- Every missing proof item becomes a launch blocker or provisional label, not a silent gap.

## Operating Rule

The sprint ends when proof records exist. Drafting starts after proof, not before it.
