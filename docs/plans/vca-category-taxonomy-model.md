# VibeCode Authority Category Taxonomy Model

## Principle

VibeCode Authority needs controlled language. The site can rank and compare tools only when terms like vibe coding, prompt-to-app, AI IDE, agentic coding, no-code incumbent, and production readiness are used consistently.

## Core Terms

| Term | Definition | Use when | Avoid when |
| --- | --- | --- | --- |
| Vibe coding | Building software through natural-language prompts, iterative AI edits, and fast feedback loops instead of traditional planning-first development | The workflow centers on prompting, previewing, repairing, and shipping with AI assistance | The product is only a static website generator, generic chatbot, or traditional no-code workflow with minor AI features |
| Prompt-to-app | A tool class that attempts to generate a usable web app, data model, UI, and deployment path from a written prompt | The product can create multi-screen applications or functional prototypes from instructions | The product only generates UI mockups, landing pages, or isolated code snippets |
| AI IDE | A code-native development environment where AI edits, explains, debugs, or generates code inside an existing project workflow | The buyer needs repo-level changes, debugging, refactors, or production code stewardship | The buyer expects no-code app creation without reading or editing code |
| Agentic coding | A workflow where an AI system plans and executes multi-step software tasks across files, commands, tests, and deployment checks | The tool can take delegated implementation tasks and operate across a codebase | The tool is limited to autocomplete, chat suggestions, or isolated code generation |
| No-code incumbent | A mature visual platform that predates the vibe-coding wave and now adds AI assistance to structured app or site building | Comparing control, maturity, integrations, and lock-in against faster AI-native builders | Describing a new AI-first prompt-to-app product |
| Production readiness | The degree to which a generated app can be deployed, secured, maintained, handed off, and trusted beyond a demo | Discussing auth, data rules, secrets, performance, export path, ownership, and maintainability | The only evidence is a working preview or attractive screenshot |

## Boundary Rules

- Do not call a tool no-code if the target user must inspect, edit, or deploy code to succeed.
- Do not compare AI IDEs directly against prompt-to-app tools without stating buyer intent.
- Do not use production-ready unless deploy, auth, data, secrets, and handoff checks have evidence.
- Do not call a UI generator an app builder unless it produces functional application behavior.
- Do not let SEO keywords override the category definition used in the verdict.

## SEO Usage Rules

| Page type | Rule |
| --- | --- |
| Review pages | Use the tool's true category in the intro, verdict, schema, and internal links |
| Comparison pages | State whether the comparison is direct, adjacent, or buyer-intent based |
| Best-of pages | Group rankings by use case before declaring broad category winners |
| Reports | Define the evidence window and category scope before presenting rankings |

## Operating Outcome

This model keeps authority language consistent as the site scales. It prevents the ecosystem from becoming a keyword soup where prompt-to-app tools, AI IDEs, UI generators, and traditional no-code platforms are treated as interchangeable.
